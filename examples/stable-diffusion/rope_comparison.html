B. interleaved reshape(-1,2).unbind(-1) [c0,c0,c1,c1,c2,c2,c3,c3] QwenImage, Flux NEEDED
Current Problem: When compiling pattern B, the compiler's contrib_split_shuffle does a half-split (first 64 / last 64) instead of the required even/odd de-interleave. This silently produces wrong results (pearsonr ≈ -0.04 after RoPE).
Note: The two patterns are NOT interchangeable. Using rotate_half on a model trained with interleaved RoPE gives pearsonr = 0.9987 and max abs diff = 11.1 on CPU (before any compilation).

Request to Compiler Team

Pattern B (reshape(-1, 2).unbind(-1)) needs to produce an even/odd de-interleave, not a half-split.

Concretely, for x with head_dim=128 stored as [2, 64] blocks in NCW64c:
• Current behavior (contrib_split_shuffle): output_0 = x[0:64], output_1 = x[64:127]
• Required behavior: output_0 = x[0,2,4,...,126] (even), output_1 = x[1,3,5,...,127] (odd)

Similarly, torch.stack([a, b], dim=-1).flatten(-2) needs to re-interleave: produce [a0,b0,a1,b1,...] not [a0,...,a63,b0,...,b63].

The existing contrib_rotary_embedding kernel may already support this via the shuffle flag — the issue is in the pattern matching / data reshaping that feeds into the kernel.
